# 开放扩展接口规范总则

## EXT-000 文档目的

本文定义“不一样的日记”开放扩展契约的产品边界、规范写法、版本规则和初始接口分组。目标读者同时包括：

- 想实现兼容服务器的人类开发者。
- 负责生成、审查或测试实现的 AI Agent。
- 维护核心客户端和一致性测试的项目贡献者。

本文是设计总则，不代表生产服务器已经实现。当前官方核心仍只实现离线与 BYOK 两条路径。

## EXT-001 核心承诺

以下要求是所有后续扩展规范的上位约束：

- `CORE-001 MUST`：没有服务器、账户和模型 Key 时，基础陪伴与本地记录必须完整可用。
- `CORE-002 MUST`：用户配置模型 Key 后，官方 AI Provider 必须从客户端直连供应商，不经过项目自有服务器。
- `CORE-003 MUST`：账户、认证、权限、同步和云存储不得成为应用启动依赖。
- `CORE-004 MUST`：未配置兼容服务器时，客户端不得展示登录或同步入口，不得发送相关探测请求。
- `CORE-005 MUST`：兼容服务器不得接收、保存、同步或记录用户的模型 Key。
- `CORE-006 MUST`：兼容服务器不得改变非审判产品规则，不得生成连续打卡、羞辱、补偿节食或补偿运动要求。
- `CORE-007 MUST`：社区实现可以使用任意技术栈，但不得要求修改核心本地数据才能完成离线使用。
- `CORE-008 MUST`：服务器故障、认证失效或同步冲突不得阻断本地记录和 BYOK AI。

任何违反 `CORE-*` 要求的实现都不能声明为兼容服务器。

## EXT-002 规范关键词

后续规范统一使用以下关键词。英文关键词必须保留，中文解释用于帮助人类阅读：

| 关键词 | 含义 |
| --- | --- |
| `MUST` | 必须满足；不满足即不兼容 |
| `MUST NOT` | 明确禁止；出现即不兼容 |
| `SHOULD` | 默认应满足；偏离时必须记录理由 |
| `SHOULD NOT` | 默认不应采用；采用时必须记录理由和风险 |
| `MAY` | 可选能力；缺失不影响基础兼容 |

AI Agent 不得把 `SHOULD` 自动提升为 `MUST`，也不得把文档未定义的行为推断为允许。

## EXT-003 双读文档格式

每份正式接口规范必须同时包含人类说明和机器契约。

### EXT-003-A 人类可读部分

每个能力必须按固定顺序写明：

1. 稳定的规范 ID，如 `AUTH-001`。
2. 目的与用户价值。
3. 范围与明确不做。
4. 前置条件和授权要求。
5. 输入字段、类型、必填性和语义。
6. 输出字段、类型和语义。
7. 稳定错误码及恢复方式。
8. 隐私、安全和禁止行为。
9. 至少一个成功示例和一个失败示例。
10. Given / When / Then 一致性用例。

说明文字必须使用明确名词，避免“它”“这个”“适当处理”等无法定位的指代。时间必须说明时区和格式，ID 必须说明生成方，空值必须说明是缺失、`null` 还是空字符串。

### EXT-003-B 机器可读部分

每份正式接口规范必须提供：

- OpenAPI：HTTP 方法、路径、鉴权、状态码、请求与响应引用。
- JSON Schema：能力文档、实体、错误体和同步信封。
- 无注释的有效 JSON 示例。
- 可自动执行的一致性测试清单。
- 语义化版本号和变更记录。

Markdown 与机器契约发生冲突时，字段结构和类型以 JSON Schema/OpenAPI 为准；产品边界和禁止行为以带稳定 ID 的规范条款为准。冲突本身必须作为规范缺陷修复，不能让实现者自行选择。

## EXT-004 目标目录结构

开放契约进入实施阶段后，使用以下结构：

```text
specs/
  SPEC_INDEX.yaml
  openapi.yaml
  schemas/
    capabilities.schema.json
    auth.schema.json
    sync.schema.json
    storage.schema.json
    error.schema.json
  docs/
    auth.md
    permissions.md
    sync.md
    cloud-storage.md
    account.md
  examples/
    capabilities.success.json
    sync.push.success.json
    sync.conflict.json
  conformance/
    manifest.yaml
    cases/
```

`SPEC_INDEX.yaml` 是人类与 AI Agent 的统一入口，至少包含每份规范的 ID、版本、状态、依赖和文件路径。AI Agent 必须先读取索引，再读取任务涉及的具体规范，不得扫描示例后自行反推协议。

## EXT-005 能力分组

### EXT-005-A Capability Discovery

用途：客户端在用户主动配置服务器地址后，确认该服务器支持哪些开放能力和规范版本。

建议端点：

```text
GET /v1/capabilities
```

初始能力文档：

```json
{
  "specId": "different-diary.extensions",
  "specVersion": "0.1.0",
  "serverName": "Example Compatible Server",
  "capabilities": {
    "auth": ["magic_link", "passkey"],
    "permissions": "owner_only",
    "sync": {
      "checkIns": true,
      "preferences": false,
      "attachments": false
    },
    "cloudStorage": false,
    "accountExport": true,
    "accountDeletion": true
  }
}
```

- `CAP-001 MUST`：客户端只能在用户主动提交服务器地址后请求能力文档。
- `CAP-002 MUST`：客户端不得因某项可选能力为 `false` 而降级本地功能。
- `CAP-003 MUST`：服务器必须明确返回支持的规范版本，不得用产品版本代替。
- `CAP-004 MUST`：未知能力字段必须被客户端安全忽略，不得自动启用功能。

### EXT-005-B Authentication Adapter

用途：为兼容服务器建立可撤销的用户会话。

初始适配器形状：

```ts
interface AuthAdapter {
  getSession(): Promise<AuthSession | null>;
  startSignIn(input: SignInInput): Promise<SignInChallenge>;
  completeSignIn(input: SignInCompletion): Promise<AuthSession>;
  signOut(): Promise<void>;
}
```

- `AUTH-001 MUST`：登录必须是用户主动动作，不能在首次打开时自动触发。
- `AUTH-002 MUST`：登录失败不得删除、锁定或隐藏本机记录。
- `AUTH-003 MUST`：会话只授予服务器资源访问权，不改变本机功能资格。
- `AUTH-004 MUST`：规范必须支持会话撤销和过期状态。
- `AUTH-005 MUST NOT`：认证请求不得包含模型 Key、用户照片或未获授权的本机记录。

邮箱魔法链接与通行密钥是可选方式；兼容服务器至少实现一种，客户端根据能力文档展示。

### EXT-005-C Permission Policy

第一版兼容模型只定义资源所有权，不设计组织、团队或企业 RBAC。

- `PERM-001 MUST`：每条云端用户数据必须绑定唯一所有者主体。
- `PERM-002 MUST`：普通用户只能读取、写入、导出和删除自己的资源。
- `PERM-003 MUST`：服务端管理权限不得通过普通客户端能力文档暴露给用户会话。
- `PERM-004 MUST`：授权失败统一返回稳定错误码，不得用空数据伪装成功。
- `PERM-005 MUST NOT`：服务器不得基于体重、饮食、运动状态或使用频率改变权限。

### EXT-005-D Sync Adapter

用途：在用户主动启用后，同步被明确允许的数据。

初始同步实体只考虑 `check_in`。照片、偏好和其他实体必须通过后续规范版本加入。

```ts
type SyncEnvelope<T> = {
  entityType: "check_in";
  entityId: string;
  schemaVersion: number;
  clientUpdatedAt: string;
  deletedAt: string | null;
  payload: T | null;
};
```

- `SYNC-001 MUST`：同步默认关闭，必须由用户主动启用。
- `SYNC-002 MUST`：批次写入必须支持幂等键，重试不得产生重复记录。
- `SYNC-003 MUST`：冲突时必须保留双方内容或生成冲突副本，不得静默覆盖。
- `SYNC-004 MUST`：删除必须可传播，并在所有活跃设备确认后不可再次读取。
- `SYNC-005 MUST`：模型 Key 永远不属于同步实体。
- `SYNC-006 MUST`：客户端断网时继续写入本机队列，不把网络失败描述为用户失败。
- `SYNC-007 MUST`：服务端无法理解更高 `schemaVersion` 时必须明确拒绝，不得截断未知字段后保存。

### EXT-005-E Cloud Storage Adapter

用途：可选保存用户明确允许上传的附件。该能力不属于离线与 BYOK 核心。

- `STORE-001 MUST`：上传前必须展示目标服务器、数据类型和用途。
- `STORE-002 MUST`：只记录模式不得调用云存储接口。
- `STORE-003 MUST`：附件删除必须同时删除原文件、派生预览和可恢复副本，法律要求保留时必须事先说明。
- `STORE-004 MUST`：下载地址必须有时效或访问控制，不得默认公开。
- `STORE-005 MUST`：服务器不得把附件用于训练、广告或未声明的分析。

### EXT-005-F Account Lifecycle

用途：允许用户带走或彻底删除兼容服务器中的数据。

- `ACCOUNT-001 MUST`：服务器必须提供结构化数据导出。
- `ACCOUNT-002 MUST`：服务器必须提供账户与同步副本删除能力。
- `ACCOUNT-003 MUST`：账户删除不得清空用户设备上的本机记录，除非用户另行在本机确认。
- `ACCOUNT-004 MUST`：删除完成后必须返回可验证的完成状态；异步删除必须提供状态查询。
- `ACCOUNT-005 MUST`：服务端导出格式必须声明 schema 版本，并能映射回核心本地导出格式。

## EXT-006 稳定错误体

所有接口使用同一错误结构：

```json
{
  "error": {
    "code": "SYNC_SCHEMA_UNSUPPORTED",
    "message": "The server does not support this schema version.",
    "retryable": false,
    "details": {
      "supportedSchemaVersions": [1]
    }
  }
}
```

- `ERROR-001 MUST`：`code` 是稳定、可枚举的机器标识。
- `ERROR-002 MUST`：`message` 面向开发者，不得直接作为最终用户文案展示。
- `ERROR-003 MUST`：`retryable` 必须明确说明原请求是否可以安全重试。
- `ERROR-004 SHOULD`：`details` 只包含恢复所需的最少信息，不得泄露其他用户、内部堆栈或密钥。

最终用户文案由核心客户端根据错误码生成，并继续遵守非审判语气。

## EXT-007 一致性用例格式

每条 `MUST` 或 `MUST NOT` 要求至少对应一条可执行或可人工复核的一致性用例。

示例：

```yaml
id: SYNC-003-conflict-preserves-both
requirement: SYNC-003
given:
  - device_a_has_revision: 4
  - device_b_has_revision: 4
  - payloads_are_different: true
when:
  - both_devices_push_without_pulling
then:
  - no_payload_is_silently_discarded
  - conflict_is_returned_or_conflict_copy_is_created
```

一致性清单必须说明：

- 测试 ID 和对应规范 ID。
- 前置服务器能力。
- 请求夹具。
- 预期状态码和响应 schema。
- 预期持久化结果。
- 是否允许重试。
- 清理步骤。

## EXT-008 AI Agent 工作规则

AI Agent 实现或审查兼容服务时必须遵守：

1. 先读取 `SPEC_INDEX.yaml`，确认规范版本和依赖。
2. 只从 OpenAPI/JSON Schema 生成字段与类型，不根据示例添加字段。
3. 逐条关联实现代码、测试和稳定规范 ID。
4. 不确定时报告规范缺口，不自行创造协议语义。
5. 运行一致性测试，并报告通过、失败和未实现能力。
6. 不把 Mock 通过等同于安全审计或生产就绪。
7. 不修改核心离线能力来迁就服务器实现。

AI Agent 的完成报告至少包含：规范版本、实现能力列表、未实现能力、失败用例、数据迁移影响和安全限制。

## EXT-009 版本与兼容

规范使用语义化版本：

- Patch：不改变语义的澄清、示例或错误修正。
- Minor：向后兼容的新字段、新能力或新错误码。
- Major：删除字段、改变语义、改变授权或破坏兼容的修改。

- `VERSION-001 MUST`：服务器能力文档必须返回完整规范版本。
- `VERSION-002 MUST`：客户端不得向不支持的 Major 版本发送用户数据。
- `VERSION-003 MUST`：新增 JSON 字段默认必须可忽略，除非 schema 明确标记为必需。
- `VERSION-004 MUST`：废弃能力至少保留一个 Minor 版本，并在变更记录中说明替代方案。

## EXT-010 规范发布门槛

单个扩展能力只有同时满足以下条件，才能从 `draft` 进入 `stable`：

- 人类说明包含完整的范围、输入、输出、错误和禁止行为。
- OpenAPI 与 JSON Schema 可通过验证。
- 所有 JSON/YAML 示例可解析。
- 每条 `MUST` / `MUST NOT` 都有一致性用例。
- Mock 参考实现通过全部必需用例。
- 核心客户端在未配置服务器时的零请求测试仍通过。
- 离线和 BYOK 路径未产生新的服务器依赖。
- 安全与隐私限制经过人工审查。

## EXT-011 当前不做

本总则当前不交付：

- 生产服务器。
- 官方托管账户或数据库。
- 组织、团队、管理员控制台或企业 RBAC。
- 计费、订阅、广告、分析或增长接口。
- 服务端模型代理或共享模型额度。
- 跨项目单点登录。

这些内容不能通过实现细节绕过规范评审后悄悄进入核心产品。
