# 技术路线图

## 0. 当前实现速览

本文同时记录“已经落地的实现”和“后续演进方向”。判断当前仓库时，以本节、代码、自动化测试和 README 的“当前状态”为准；后文未标为已完成的 Provider、账户、同步、原生封装与长期 Key 保存均属于计划，不是当前能力。

| 层级 | 当前实现（2026-07-11） |
| --- | --- |
| 应用形态 | 可安装 PWA；同时生成一个可下载的单文件 HTML |
| 前端 | React 19 + TypeScript + Vite 8 |
| 页面结构 | `app/page.tsx` 仅保留入口；`app/diary/` 分离控制器、文案模型、首页、对话页和本机记录页 |
| 本地数据 | IndexedDB + Dexie，保存匿名身份、文字、照片附件和运动详情 |
| PWA | Service Worker + Web App Manifest，同源应用壳缓存 |
| 测试 | Node Test Runner + `tsx` + `fake-indexeddb`；界面流程另做真实浏览器验收 |
| AI | 只有用户主动进入并确认后才调用 OpenRouter 文字接口；Key 只在当前页面内存中存在 |
| 发布 | Vite 静态构建由 GitHub Actions 发布到 GitHub Pages；同一次构建生成下载版 HTML |

开发、测试和公开发布现在共用同一条 Vite 静态应用链路。`npm run dev` 用于本机开发，`npm run build` 验证通用静态构建，`npm run build:pages` 额外加入 GitHub Pages 的仓库路径并生成单文件下载版。仓库不包含必须部署的 Worker、数据库、云存储或旧 Sites 配置。

## 1. 技术目标

技术必须服务于三个产品目标：

1. 无账户、无资料、无模型 Key 也能使用。
2. 数据默认保留在用户设备上。
3. BYOK AI 可以后加；登录、权限、同步和云存储只作为开放扩展契约，不阻塞核心产品。

因此采用“本地优先的可安装 Web App”，而不是第一天就建设完整后端和原生双端。

## 2. 当前选择与演进方向

| 层级 | 第一选择 | 原因 |
| --- | --- | --- |
| 应用形态 | PWA | 一个链接覆盖 iPhone、Android 和桌面，可添加到主屏幕 |
| 前端 | React + TypeScript + Vite | 当前已经落地；生态成熟，便于开源贡献和后续封装原生 App |
| 本地数据 | IndexedDB + Dexie | 能保存结构化记录、偏好和本地图片索引，支持版本迁移 |
| 状态管理 | 轻量状态层 | 只管理当前交互状态，长期数据统一进入本地数据库 |
| PWA 能力 | Service Worker + Web App Manifest | 离线打开、主屏幕图标、资源缓存和版本更新 |
| 测试 | Node Test Runner + `tsx` + `fake-indexeddb` | 当前自动验证实现、数据迁移和产品边界；浏览器流程用于发布前验收 |
| AI 接入 | Provider Adapter | 当前只有 OpenRouter 文字适配器；图片理解和其他 Provider 属于后续路线 |
| 开放扩展 | Auth / Sync / Storage Adapter | 核心只保留版本化契约，可由社区接入自托管服务或托管数据库 |
| 原生封装 | Capacitor（后期） | 复用 PWA 代码生成 Android/iOS 容器 |

不在第一阶段引入大型微服务、消息队列、复杂推荐系统或自建模型。

## 3. 数据原则

### 本机模式是完整产品

首次启动生成随机本机用户 ID。当前进入 IndexedDB 的数据包括：

- 文字记录
- 经本机处理的照片附件
- 可选运动详情
- 回应历史

用户偏好、可选身体资料和模型设置仍属于后续数据模型；当前版本不会为了这些计划字段提前询问用户或持久化模型 Key。

本机模式不是试用版，也不通过功能阉割强迫用户注册。

### 登录和云端只属于可选扩展

官方核心不建设必须部署的账户和同步服务器。兼容服务器可以根据开放契约建立“本机 ID 与云端账户”的映射，再进行可撤销的数据合并；同步冲突必须优先保留两份，不得静默覆盖用户记录。

未配置兼容服务器地址时：

- 不展示登录、权限或同步入口。
- 不发送账户、记录或设备标识相关请求。
- 不影响离线记录、数据导出和 BYOK AI。

### 用户拥有数据

必须支持：

- 导出为 JSON，后续可增加 Markdown。
- 删除单条记录。
- 清空本机数据。
- 配置兼容服务器后，删除该服务器中的账户及同步副本。

## 4. AI 接入路线

### 基础体验不依赖模型

本地回应引擎由以下部分组成：

- 经人工审核的回应库。
- 当前状态规则。
- 用户明确保存的偏好。
- 随机变化与近期去重。

即使断网、免费额度耗尽或模型服务故障，核心流程仍可使用。

### 模型接口统一

当前文字 AI 使用的统一接口是：

```ts
interface AIProvider {
  readonly id: string;
  validateKey(key: string): Promise<void>;
  respond(key: string, input: CompanionInput): Promise<AIProviderResponse>;
}
```

当前与计划适配器：

- OpenRouter Provider：已经实现文字 Key 验证、文字回应、错误降级和非审判规则检查。
- Qwen 或其他视觉 Provider：计划项，只有完成图片发送确认和隐私边界后才会接入。
- 测试替身：自动测试通过可注入请求函数验证，不把 Mock 暴露为用户功能。

免费模型和额度变化很快，具体模型名称放在远端可更新配置或本地配置文件中，不写死在核心业务逻辑里。

### Key 的处理

- 当前实现：不保存 Key，只在当前页面内存中使用；离开或刷新后需要重新连接。
- 后续计划：只有用户主动选择“记住在本机”时，才研究使用 Web Crypto 加密保存，并补齐威胁模型与迁移测试。
- Key 不参与同步，不进入日志，不发送到自有分析服务。
- 页面不加载不必要的第三方脚本，降低浏览器存储被读取的风险。
- 必须明确告知：浏览器本地存储无法抵御设备失窃或恶意脚本等全部风险。

若某个模型服务不允许浏览器直连，则不进入官方 BYOK 直连适配器。社区可以另行实现代理扩展，但代理不属于核心兼容要求，也不得让离线或其他直连 Provider 依赖它。

## 5. 图片处理路线

图片默认先在设备端处理：

1. 用户选择或拍摄照片。
2. 去除 EXIF 等元数据。
3. 缩小到模型理解所需尺寸。
4. 生成本地预览。
5. 明确提示将发送给哪一家模型服务。
6. 仅在用户选择 AI 理解时发送。
7. 默认不上传到自有服务器。

“只记录”模式不调用任何模型，也不把照片传出设备。

## 6. 分阶段实施

### 阶段 0：交互原型

交付：

- 手机尺寸的可点击首页。
- 四条核心分支。
- 一组经过人工确认的回应文案。
- 无需数据库和真实 AI。

通过条件：在真机上走通后，用户能感受到“不审判”，而不是只看到产品说明。

当前状态（2026-07-10）：已验收通过。四个核心入口及各自的“先回应、再选择、再确认”点击闭环已实现，手机窄屏可直接使用。所有选择仅保留在当前页面状态中，刷新即清空；数据库、登录、真实 AI、图片识别和云同步仍明确留在后续阶段。构建、自动化边界检查和用户体验验收均已通过。

### 阶段 1：本地优先 MVP

交付：

- PWA 安装能力。
- IndexedDB 数据模型。
- 本机匿名用户。
- 文字记录与时间线。
- 本地回应引擎。
- 数据导出和清空。
- 离线基础体验。

通过条件：断网、未登录、未填写身高体重、未连接 Key 时仍能完整使用。

当前状态（2026-07-10）：已验收通过。IndexedDB + Dexie v1、本机匿名身份、四分支本地回应轮换、文字记录、时间线、单条删除、JSON 导出、清空并重建身份、PWA Manifest、离线应用壳和暖色编辑式视觉均已完成；31 项阶段 1 自动化测试和用户主观体验验收通过。实施证据见 [`04-阶段1设计.md`](04-阶段1设计.md)。

### 阶段 2：本机照片记录

交付：

- 设备端图片压缩和元数据移除。
- IndexedDB v2 附件表。
- “只记录”零网络保存。
- 时间线回看与级联删除。

通过条件：不连接模型时，照片仍能在本机保存、回看和删除；原图元数据不进入存储。

当前状态（2026-07-10）：已完成并进入下一阶段，包括 Dexie v2 附件、设备端重新编码、只记录零网络、时间线回看和级联删除；容量恢复、照片 ZIP 导出和不同设备兼容性复核作为后续维护项保留。完整设计与稳定验收 ID 见 [`06-阶段2设计.md`](06-阶段2设计.md)。

### 阶段 3：本机运动记录

交付：

- IndexedDB v3 运动详情表和 v1/v2 原地迁移。
- 首页辅助入口与“刚练完”分支的同一记录页。
- 全部可选的活动类型、时长、步数、距离和备注。
- 时间线、详情、级联删除、清空和 JSON v3 导出。
- 独立的界面、展示、本机写入与导出职责单元。
- 保存后独立奖励组件与纯 CSS 大动画；减少动态效果时静态呈现。

通过条件：全部详情跳过也能保存；升级不丢旧文字或照片记录；没有网络、模型、账户、评分和补偿逻辑。

当前状态（2026-07-11）：记录闭环、保存后奖励页与基于今日本机记录的 Canvas 海报均已由用户完成手机端主观体验验收；只读本机运动图表作为后续维护切片保留。运动设计与验收 ID 见 [`07-阶段3运动记录设计.md`](07-阶段3运动记录设计.md)，海报边界见 [`09-本机今日海报设计.md`](09-本机今日海报设计.md)。

### 阶段 4：BYOK AI

交付：

- 至少一个通过浏览器直连验证的真实 Provider；Mock 只用于自动测试。
- Key 验证、会话保存和可选本机加密保存。
- 用户意图驱动的图片理解和明确发送确认。
- 用户主动询问时，对运动消耗给出宽松范围和不确定性说明。
- 模型故障降级到本地回应，本机记录不丢失。

通过条件：Provider 不可用或 Key 错误时不丢记录、不阻断用户、不出现技术性责备；未连接 Key 时本机体验仍完整。

开始条件：阶段 3 已由用户验收通过。

当前状态（2026-07-11）：开始条件已满足，第一切片已通过用户体验验收。OpenRouter 浏览器直连、会话级 Key 验证、用户主动发送的文字回应、非审判规则检查、Provider 失败时的本机降级，以及 AI 回应由用户选择后保存到本机均已完成；Key 本身不会持久化。图片理解、运动消耗估算和 Key 加密长期保存仍未接入。设计与验收边界见 [`08-阶段4可选AI设计.md`](08-阶段4可选AI设计.md)。

### 阶段 5：开放服务器契约

交付：

- 兼容服务器能力发现规范。
- Auth、Permission、Sync、Cloud Storage 和 Account Adapter 契约。
- OpenAPI、JSON Schema、稳定错误码和版本策略。
- 可供人类与 AI Agent 阅读的规范说明和有效示例。
- Mock 服务及一致性测试套件。

通过条件：第三方无需阅读核心内部实现即可开发兼容服务器，并通过一致性测试；未配置或未连接服务器时，本机模式与 BYOK 模式仍为一等公民。

官方核心在这一阶段不交付生产账户系统、云数据库或托管同步服务。规范格式与边界见 [`05-开放扩展接口规范.md`](05-开放扩展接口规范.md)。

### 阶段 6：安装包与商店

交付：

- 使用 Capacitor 封装 Android。
- GitHub Releases 自动发布签名 APK。
- 根据真实留存决定是否加入 TestFlight 和 App Store。

通过条件：Web、Android 和 iOS 共用核心规则与数据模型，不产生三套产品逻辑。

## 7. 质量门槛

每个阶段至少验证：

- 手机窄屏和大字号可用。
- 刷新、离线、升级后本地记录不丢失。
- 所有可选步骤都确实可以跳过。
- 模型输出不会绕过非审判规则。
- “只记录”不会意外触发网络请求。
- 删除本机数据后不可再次读取；兼容服务器还必须通过云端删除一致性测试。
- 不存在连续打卡、超标红色警告或补偿性建议的文案回归。

## 8. 当前推荐顺序

不要先做登录、云数据库或模型接入。第一轮开发顺序为：

1. 真机交互原型。
2. 本地数据与无 Key 体验。
3. 本机照片记录。
4. 本机运动记录并完成人工验收。
5. BYOK AI。
6. 开放扩展契约与一致性测试。
7. 原生安装包。

这样最先验证的是产品是否真的能接住人，而不是技术系统能否注册账户和调用模型。
