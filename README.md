# 减肥拍拍乐

一个面向“同时上班、控制体重、偶尔健身”的普通人的非审判型陪伴工具。

## 当前进度

阶段 0 手机端交互原型已于 2026-07-10 验收通过，已完成四个核心入口及各自的可点击回应闭环：

- 我想吃点东西
- 我今天不想练
- 我刚练完，很累
- 没什么，只是来坐坐

阶段 1“本地优先 MVP”已于 2026-07-10 验收通过。最终版在不恢复大圆角卡片的前提下，以 Paper、Cafe、Spacious 规则完成暖色编辑式返工。设计与验收记录见 [`docs/04-阶段1设计.md`](docs/04-阶段1设计.md)。

阶段 1 最终版已实现本机匿名身份、80 条人工回应、可选文字记录、时间线、单条删除、JSON 导出、彻底清空、PWA 安装和离线应用壳。视觉层使用暖纸底色、咖啡棕文字、低饱和状态色、缓慢环境光和页面节奏，同时将四入口、选择、回应、时间线与数据管理改为中文序号目录、引文和渐隐细线列表，不再依赖重复圆角卡片。数据刷新后仍保留在当前设备。

阶段 2“本机照片记录”已完成：照片在设备端重新编码后才保存；“只记录”路径不识别、不发送，时间线回看和级联删除已接通。照片 ZIP 导出仍作为后续维护项，设计边界见 [`docs/06-阶段2设计.md`](docs/06-阶段2设计.md)。

阶段 3“本机运动记录”已于 2026-07-11 通过用户体验验收。首页辅助入口和“刚练完”分支都能进入同一记录页；活动类型、时长、步数、距离和备注全部可以跳过，保存后会进入专属大动画奖励页，再进入本机时间线、详情和 JSON 导出。运动记录不计算完成度、连续天数或好坏，职责边界与验收项见 [`docs/07-阶段3运动记录设计.md`](docs/07-阶段3运动记录设计.md)。

阶段 4“可选 AI”已进入第一切片：用户可以从本机回应页主动连接自己的 OpenRouter Key，在明确发送确认后，让 AI 回应刚写下的文字。Key 只在当前页面内存中使用，不写入记录或导出；照片、历史记录和身体资料不会随第一切片请求发送。模型失败或回应越过非审判规则时，页面保留用户输入并退回本机人工回应。图片理解、运动消耗估算和 Key 加密长期保存留在阶段 4 后续切片，设计边界见 [`docs/08-阶段4可选AI设计.md`](docs/08-阶段4可选AI设计.md)。

## 开源实现原则

官方核心只实现两条不依赖自有服务器的路径：完全离线使用，以及用户自带 Key 使用。账户、认证、权限、云存储和同步作为可选开放扩展，不成为应用启动或核心体验的依赖。

项目会维护版本化的开放扩展契约、OpenAPI/JSON Schema 和一致性测试，方便社区开发者或 AI Agent 实现兼容服务；官方核心不预置一套必须部署的业务后端。规范总则见 [`docs/05-开放扩展接口规范.md`](docs/05-开放扩展接口规范.md)。
